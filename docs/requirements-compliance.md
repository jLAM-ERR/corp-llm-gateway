## Ключевой архитектурный вывод

**В сервисе нет хардкод-детекторов ПДн/секретов.** Вся анонимизация (пункты 4–9) делегирована:

1. **corp-LLM как оракул** — system prompt + tool `report_pii_findings` инструктируют внутреннюю модель найти «all PII or corp-confidential terms» (`sanitizer/orchestrator.py`).
2. **Командные правила `replace.md`** — литеральные пары `паттерн → плейсхолдер`, которые **переопределяют** суждение модели (`rules/parser.py`, `docs/replace-md-authoring.md`).

Детекторы `Presidio` и `OpenaiPrivacyFilter` — заглушки (`NotImplementedError`).

**Следствие для compliance:** покрытие требований 4–9 держится на (а) качестве суждения внутренней LLM и (б) полноте правил `replace.md`, которые операторы пишут вручную. Категории «реквизиты контрагентов» (5), «ключевые продукты» (8) и «ПОД/ФТ» (9) сегодня **не имеют ни встроенной логики, ни готовых правил** — это самый существенный разрыв для аудита.

Что усиливает защиту независимо от модели:
- **Инвариант «оригиналы не утекают»** (M1-14, `tests/invariants/test_no_originals_leak.py`) — 6 поверхностей.
- **Гейт NEVER-полей** (`audit/invariants.py` + Vector VRL) — аудит-запись с оригиналами/секретами отклоняется.
- **Fail-closed** при отказе corp-LLM / Redis / Postgres / S3 / Vector-буфера (матрица M4).

## Жёлтые пункты — что нужно для «зелёного»

- **1 (auth пользователей):** переключить провайдер с `noop` на реальный (`bearer`/`oidc`) + вынести токены из in-memory в Postgres. Код enforcement готов, не хватает конфигурации и persistent-хранилища.
- **4 (ПДн):** добавить allowlist/исключение тестовых данных (сейчас его нет) — иначе требование «кроме тестовых» формально не выполнено.
- **5/6/7/8 (категории анонимизации):** поставить встроенные детекторы или хотя бы базовый набор регекспов (ИНН/КПП/БИК, IPv4/hostname, JWT/Bearer/PEM) как «пол» под суждение LLM, не полагаясь только на ручной `replace.md`.
- **12 (DLP):** включить `NetworkPolicy` + CoreDNS-sinkhole в проде (Helm-шаблоны готовы, `enabled: false`).
