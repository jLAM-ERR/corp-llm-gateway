# Руководство по составлению `replace.md`

Ссылка на план: M8-4.

`replace.md` — это набор правил санитизации на уровне команды. Корп-LLM
применяет эти правила, чтобы найти и заменить специфичные для команды
термины до того, как запрос покинет корп-периметр.

## Расположение файла

По умолчанию шлюз ищет файл `<rules-dir>/<team-id>.md`. Путь
настраивается в `team_config.replace_md_path`.

## Формат

Одно правило на строку. Грамматика строгая — некорректные строки
отклоняются с указанием номера строки.

```
- ORIGINAL → REPLACEMENT
```

Замечания:

- Разделитель — стрелка `→` (U+2192), а **не** ASCII `->`.
  ASCII `->` отклоняется явно, чтобы два визуально похожих формата
  не могли сосуществовать.
- ORIGINAL и REPLACEMENT можно обернуть в обратные кавычки для наглядности:

  ```
  - `Project Polaris` → `[CONFIDENTIAL_PROJECT]`
  ```

- Строки, начинающиеся с `#`, — комментарии. Как и строки, начинающиеся
  с `<!--` (HTML-стиль для редакторов, отображающих markdown).
- Пустые строки игнорируются.
- Пробелы вокруг токенов обрезаются.

## Пример файла

```markdown
# Team X replace.md
# Owner: alice@corp.lan

- `Project Polaris` → `[CONFIDENTIAL_PROJECT]`
- `Acme-Internal-CRM` → `[INTERNAL_TOOL]`
- `dr.smith@partnerlab.com` → `[PARTNER_CONTACT]`
- `BadgeID-XYZ-12345` → `[BADGE_ID]`

# Hostnames
- `db-prod-13.corp.internal` → `[INTERNAL_HOST]`
- `db-prod-14.corp.internal` → `[INTERNAL_HOST]`
```

## Обновления на лету

Согласно M1-15: обновления правил вступают в силу при вытеснении из кэша
(по умолчанию 5 мин). Активные диалоги, удерживающие маппинги до
обновления, продолжают их использовать до истечения диалога; новые
вхождения в том же диалоге подхватывают новые правила.

Принудительно обновить можно, уменьшив cache TTL команды или перезапустив
один pod шлюза (rolling restart подхватывает новые правила без потери
трафика).

## Советы по составлению

- **Будьте конкретны**. `- foo → [BAR]` заменит каждое `foo` в каждом
  запросе. Если `foo` легитимно встречается во многих контекстах, замена
  их сломает.
- **Порядок не влияет на корректность**, но движок сортирует паттерны по
  убыванию длины перед заменой (инвариант M1-9), чтобы предотвратить
  затенение.
- **Пустые ORIGINAL или REPLACEMENT отклоняются** — парсер выдаёт ошибку
  на этапе загрузки.
- **Сначала тестируйте в staging**. Примените правила в
  `gateway-staging.corp.lan` и прогоните тестовый запрос, чтобы
  подтвердить поведение перед промоутом.

## Частые ошибки

| Ошибка | Эффект | Исправление |
|---|---|---|
| Использован `->` вместо `→` | Ошибка парсинга на этапе загрузки | Заменить на стрелку |
| Паттерн слишком короткий или общий | Много ложных срабатываний | Добавить кавычки + больше контекста |
| Несколько правил с одним паттерном | Побеждает последнее | Так не делать |
| Строка-комментарий без префикса `#` | Трактуется как правило, парсинг падает | Добавить `#` |
