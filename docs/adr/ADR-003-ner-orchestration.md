# ADR-003. Движок локального NER для пред-фильтра (RU + EN)

## Статус

**Принято.** План `docs/plans/20260630-bilingual-local-first-detection.md`, задача DP-0 (2026-07-01).

Решает «единственный гейт» перед реализацией Stage-3 (DP-2): **hand-roll dual-NER** против
**Presidio-как-оркестратора**, и выбор RU-движка (DeepPavlov против Natasha/Slovnet).

## Контекст

Подход A (local-first cascade) переносит детекцию с LLM-оракула на локальный «пол». Структурные
сущности (ИНН/КПП/ОГРН, секреты, инфраструктура) закрыты детектором regex+checksum (DP-1).
Остаётся **NER** для неструктурного: ФИО, организации, адреса (R4/R5) — и он обязан быть:

- **двуязычным (RU + EN)** — запросы регулярно смешанные (русские комментарии + английские
  идентификаторы в одном файле); конкурент `ru-llm-proxy` уже двуязычен (Presidio EN + DeepPavlov RU);
- **быстрым на CPU** — латентность это требование первого класса DRI, а корп-k8s **без GPU**
  (см. memory «no GPU»);
- встраиваемым в реестр `PIIDetector` (ADR-001) под наш placeholder-allocator (M1-9).

Вопрос: брать ли **Presidio** как фреймворк (готовые EN-recognizer'ы, мульти-движок) и какой
RU-движок ставить — тяжёлый **DeepPavlov (ruBERT)**, как у конкурента, или лёгкие
**Natasha/Slovnet**.

## Данные бенчмарка

CPU, macOS arm64, 2026-07-01. Тёплая per-call инференция (модель грузится один раз на старте
процесса), 5 фикстур RU/EN/смешанные. Recall = доля gold-сущностей (ФИО/орг/гео), найденных
движком. Это **решение-ориентированный спайк**, не полноценный F1-бенч.

| Движок | recall | p50 | p95 | Вес / зависимости | Примечание |
|---|---:|---:|---:|---|---|
| **Natasha/Slovnet (RU)** | 0.73 | **0.4 ms** | **0.7 ms** | ~30 MB, **без torch** | суб-миллисекунда на CPU |
| **spaCy `en_core_web_md` (EN)** | 0.93 | 3.0 ms | 4.1 ms | ~40 MB модель | быстрый, лёгкий |
| Presidio (EN) | — | — | — | spaCy + фреймворк | при init требует `en_core_web_lg`; без него **жёстко падает** (SystemExit мимо `except Exception`); NER внутри — тот же spaCy |
| DeepPavlov `ner_rus_bert` (RU) | н/д | н/д | н/д | torch + **1.44 GB** модель | **не устанавливается**: требует `torch>=1.6,<1.14`, а под нашу платформу таких колёс нет (доступны только ≥2.0). BERT на CPU = 100–1000× медленнее Natasha |

## Решение

**Hand-roll dual-NER**, run-both-union, за интерфейсом `PIIDetector`:

- **RU** — Natasha/Slovnet (`detectors/ner_ru.py`);
- **EN** — spaCy `en_core_web_md` (`detectors/ner_en.py`);
- объединение спанов в `detectors/dual_ner.py` (де-overlap по длиннейшему, провенанс лейблов).

**Отклонено для v1:**

- **DeepPavlov** — не ставится на современный стек (пин `torch<1.14` без колёс), тянет 1.44 GB и
  BERT-инференцию, несовместимую с требованием латентности на CPU. Это эмпирический «kill-shot»
  против выбора конкурента.
- **Presidio-как-оркестратор** — его NER это всё тот же spaCy (выигрыша в качестве нет), но он
  навязывает свою конфигурацию recognizer'ов, по умолчанию требует тяжёлую `en_core_web_lg` и
  жёстко падает без неё. Накладные расходы и потеря контроля без выгоды. (Отдельные
  regex-recognizer-идеи Presidio мы и так перекрыли детектором DP-1.)

## Обоснование

1. **Латентность (топ-требование).** Natasha 0.4 ms + spaCy 3 ms → run-both-union ≈ 3–4 ms p95.
   Это на порядки быстрее BERT-на-CPU и снимает узкое место оракула (7.4–15.6 s в бенчмарке).
2. **CPU-only.** Hand-roll стек не тянет torch/CUDA. DeepPavlov тянет torch + 1.44 GB.
3. **Контроль.** Hand-roll даёт точный run-both-union и маппинг лейблов под наш allocator;
   Presidio навязал бы свою модель и конфиг.
4. **Recall.** EN 0.93 — хорошо. RU 0.73 — приемлемо как **пол** под LLM-оракул (оракул ловит
   остаток; structured-сущности и так у DP-1). Поднять RU-recall позже можно правилами Yargy /
   дообучением Slovnet — без смены архитектуры.

## Последствия

### Положительные
- Суб-5 ms NER на CPU; лёгкие зависимости; двуязычность из коробки.
- Встраивается в реестр `PIIDetector` (ADR-001) — заглушки `PresidioDetector` /
  `OpenaiPrivacyFilterDetector` заменяются реальными `ner_ru` / `ner_en` / `dual_ner`.

### Отрицательные / стоимостные
- Держим две модели. RU-recall ниже BERT — компенсируется оракулом и regex-полом (DP-1).
- `en_core_web_md` ставится **прямым wheel-URL** (его нет в PyPI по имени) — отразить в сборке
  образа / CI, а не через `python -m spacy download` (в uv-venv нет pip).

### Нейтральные / эксплуатационные
- **Runtime Python = 3.12.** У spaCy/Natasha нет колёс под 3.14; проект и так таргетит
  `requires-python>=3.12`. Локальный `.venv` на 3.14 для NER непригоден. Решение: импорты NER —
  **ленивые**, зависимости — в **optional-extra `ner`** (DP-2), тесты NER — `importorskip`. Так
  пакет остаётся импортируемым на 3.14 (грациозная деградация), а авторитетный прогон NER-тестов —
  на 3.12.
